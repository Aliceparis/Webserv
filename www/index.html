webserv tester

Project Overview

Static files

Serve files and directories with content-type, content-length, index resolution, and autoindex.

HTTP methods

Support for GET, POST, and DELETE with route-based method restrictions.

Upload

Support for raw body upload and multipart/form-data upload under /upload/.

CGI

Execution of .sh and .py CGI scripts with environment forwarding and body piping.

Redirect

Configured 301 external redirect and 302 internal redirect through location rules.

Error pages

Custom HTML error pages mapped from configuration for major HTTP error codes.

CGI failure handling

Parent process checks CGI child termination and converts abnormal or failed CGI execution into 502.

Connection handling

Keep-alive and framed responses with explicit headers when appropriate.

Recommended Evaluation Path

  1. Static file Check /hello.txt for 200 and visible body content.
  2. Directory and index Check /dir/, /dir/index.html, and /images/.
  3. Upload Upload a file to /upload and verify that it can be read back.
  4. Delete Delete an uploaded file and verify 204 or configured delete behavior.
  5. CGI environment Use test_env.sh and test_query.sh to inspect CGI variables.
  6. CGI body Use echo_body.sh to verify POST body forwarding to CGI.
  7. CGI failure Use test_crash.sh or test_zero_div.py and confirm 502.
  8. Redirect Open external 301 and internal 302 redirect tests.
  9. Error pages Use missing resources or CGI failures to inspect mapped custom HTML error pages.

Implementation Notes

  • Request parsing is incremental, with header and body handling separated by parser state.
  • Location rules resolve static serving, CGI, upload endpoints, and redirects from configuration.
  • CGI execution is based on child process creation and pipe capture for output and request body forwarding.
  • CGI failure handling uses child process exit status and invalid CGI output detection to produce 502 Bad Gateway.
  • Custom error pages are loaded from configured error_page mappings when available.
  • Upload supports both raw body writes and multipart parsing under the upload endpoint.
  • DELETE removes files and returns the project’s configured success status behavior.
  • Redirect responses are produced from configured return rules and include a valid Location header.

Terminal Equivalents

These command-line examples provide a second way to verify the same features outside the browser tester. They are especially useful for checking raw HTTP behavior, redirect headers, CGI failures, upload responses, and delete logic.

Static files

What this tests: basic file serving, response status, headers, and body content.


      curl -i http://127.0.0.1:8080/hello.txt
      curl -i http://127.0.0.1:8080/dir/
      curl -i http://127.0.0.1:8080/not-found-file

Redirect

What this tests: configured 301 external redirect and 302 internal redirect.


      curl -i http://127.0.0.1:8080/redirection/
      curl -i http://127.0.0.1:8080/oldplace/
      curl -i -L http://127.0.0.1:8080/oldplace/

CGI

What this tests: CGI environment, query forwarding, POST body forwarding, and CGI failure handling.


      curl -i http://127.0.0.1:8080/cgi-bin/test_env.sh
      curl -i "http://127.0.0.1:8080/cgi-bin/test_query.sh?name=bonbon&x=42"
      curl -i -X POST --data "hello=world" http://127.0.0.1:8080/cgi-bin/echo_body.sh
      curl -i http://127.0.0.1:8080/cgi-bin/test_zero_div.py

Upload

What this tests: raw upload and multipart upload under the configured upload endpoint.

Stress Test

Siege command

This stress test uses siege to send many concurrent requests to the server root endpoint.

What this tests: concurrency handling, event loop stability, response throughput, and whether the server stays alive under sustained load. If siege is not installed: sudo apt update && sudo apt install siege. 


        sudo apt update
        sudo apt install siege                                              
                                                                            
siege -c 255 -t 1M http://127.0.0.1:8080/

Parameter explanation

  • siege launches the load testing tool
  • -c 255 means 255 concurrent simulated clients
  • -t 1M means the test runs for 1 minute
  • http://127.0.0.1:8080/ is the target URL

How to read the result

  • Transactions shows how many requests were completed
  • Availability shows the success rate
  • Elapsed time shows total test duration
  • Data transferred shows how much response data was served
  • Response time shows average latency
  • Transaction rate shows throughput per second
  • Concurrency shows how many clients were active on average
  • Failed transactions helps detect instability or crashes
